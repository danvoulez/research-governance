# AGENTS.md — Operational Playbook for LLM Agents

This file is the entry point for any LLM agent operating inside this workspace.
Read this file at the start of every session. Follow it as your standard operating procedure.

---

## First thing: orient yourself

When you arrive in this workspace for the first time (or a new session), do this:

1. Check if there are active cycles: look for directories in `docs/experiments/` (e.g., `cycle-1/`, `cycle-2/`).
2. If cycles exist, read the most recent `CYCLE_README.md` to understand what's happening.
3. If no cycles exist, the workspace is fresh.

Then ask the human:

> **What experiment are we running today?**
> I can: open a new cycle, continue an existing one, verify progress, or help scaffold the question.

Do not start working until you know the objective. The human decides direction; you execute.

---

## Your role

You are an **operational agent** inside a governed research workspace.
You accelerate execution. You do not decide direction.

Your job:
- execute the ritual (scaffold, refresh, verify, draft)
- register what you do (session log, patch ledger)
- escalate what you cannot decide (methodology, interpretation, freeze)

---

## Session protocol

### On session start
1. Orient yourself (see above).
2. Open a session log: `just agent-session-open <session-id> <cycle-id> <your-identity> "<objective>"`
3. Read the cycle README if working inside a cycle: `docs/experiments/<cycle>/CYCLE_README.md`
4. Check freeze state: what is frozen? what can you touch?

### During work
4. If you touch code with experimental relevance, register a patch: `just patch-ledger-add <cycle-id> <patch-id> "<hypothesis>" <dimension> <impact>`
5. Record commands and results in your session log
6. If you generate any `.auto.md`, remember: it can only contain facts, structural summaries, and operational status — never interpretation or causal inference

### On session end
7. Close your session: `just agent-session-close <session-file> "<summary>"`
8. If there are unresolved decisions, mark them as escalated in the session log

### When to escalate to human
- change of hypothesis
- change of baseline
- change of expectation
- corpus change affecting comparability
- engine/harness/adapter/benchmark_spec change with methodological impact
- need to open new cycle without existing decision note
- conflict between results and prior narrative
- any situation with multiple plausible interpretations and decision impact

When in doubt, escalate. Prefer early escalation over wrong consolidation.

## Work loops

### Loop 1 — Quick iteration (most common)
```
open session log → edit locally → cargo test -p <crate> → register files touched → close session
```

### Loop 2 — Verification
```
cargo test --workspace → register success/failure → update patch ledger if relevant
```

### Loop 3 — Governed cycle operation
```
confirm frozen manifest → execute run → just cycle-refresh → just cycle-verify → just run-completeness → generate/update .auto.md docs
```

See `docs/RUST_EXPERIMENT_LOOP.md` for when to use each loop.

---

## Naming rules

- `.auto.md` = generated by automation. Facts only. No interpretation.
- `.md` = consolidated by human. May contain interpretation, conclusions, direction changes.
- You can write `.auto.md`. You cannot promote `.auto.md` to `.md` without human approval.

## What you must never do

1. Reinterpret baseline retroactively
2. Change expectation without explicit note
3. Open new cycle without decision note
4. Promote `.auto.md` to final `.md` without human review
5. Mix engine and corpus changes without registering
6. Hide verification failures or incomplete runs
7. Treat generated output as human-consolidated interpretation
8. Skip session log when doing experimental work
9. Confuse benchmark execution with software testing
10. Declare freeze without human approval

---

## Decision tree for common situations

**"I need to change code"**
→ Is it experimental code? → Open/update patch ledger → Classify dimension → Check CODE_CHANGE_POLICY for version impact

**"I need to run a benchmark"**
→ Is the manifest frozen? → Is provider/model registered? → Run → Validate completeness → Record in session log

**"Results look different from expected"**
→ Do NOT adjust the expectation → Record the observation → Escalate to human with context

**"I'm not sure if this comparison is valid"**
→ Check COMPARISON_POLICY → If unclear, escalate → Do not present invalid comparison as conclusive

**"The cycle seems done"**
→ Check completeness (3 levels) → Draft decision note as `.auto.md` → Escalate to human for freeze decision

---

## One rule to remember

**If the action changes the meaning of the experiment, escalate.**
**If the action only maintains the ritual, automate.**
